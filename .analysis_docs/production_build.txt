1. Source Precedence and Executive Summary
 
Source-of-truth order
1) eLekha v2.1 lock list
2) current Bizflow UI repository flow and visible routes
3) previous eLekha master development document
4) country and tax appendices for Nepal, India, and global readiness

This document replaces the older mixed specification with a UI-aligned production build guide. The current repository already contains nearly all major surfaces needed for the app, but those surfaces behave as a demo shell with local state and mixed concerns rather than as final production modules. The main architectural problem is not missing screens - it is that Personal, Business, Admin, and future features are still blended together in shared pages and menus.
The production target remains one eLekha app with two top-level operating modes: Personal and Business. The user should feel like they are inside one coherent product, but the codebase, data ownership, routing, permissions, summaries, reports, and profile surfaces must stay context-aware and intentionally separated.
Decision area
Production decision
Auth
Phone number + password only in v2.1. Signup = full name + phone + password. Forgot-password remains hidden until complete. Social auth is out of launch scope.
Mode model
One app, two operating worlds: Personal and Business. Shared shell is allowed; shared data ownership is not.
Launch domains
Foundations + Transactions + Ledger + EMI & Loans core are launch-critical. POS, Products, Inventory, Billing, Reports, User Management, and Subscription can be fully documented now but shipped in later phases.
Routing
Resolve context in this order: activeProfile -> mode -> activeBusiness (if business) -> activeAccount -> route outcome.
Security
Tokens and secrets stay in secure storage only. WatermelonDB or local DB equivalents must not hold secret session values.
Data integrity
Transactions, Ledger, and EMI remain separate domains. InstallmentPaymentLink remains the installment source-of-truth.

3. Production Corrections Required Before Build
 
The product is not missing a design system or a route map. It is missing a reliable production contract between mode, ownership, routing, domain boundaries, and launch scope. The corrections below turn the current UI prototype into a scalable application specification.
Correction A - lock the two-mode model
Personal mode owns personal transactions, personal accounts, personal contacts, personal recurring money flows, and personal EMI / loans.
Business mode owns ledger, products, inventory, POS, billing, business reports, staff, roles, subscription entitlements, and business EMI / loans.
Shared modules such as Accounts, Categories, Contacts, Attachments, Reminders, and Notifications must always carry owner context.

Correction B - separate launch-critical vs later modules
Document the full app now, but treat v2.1 shipping scope as foundations + Transactions + Ledger + EMI core.
Products, Inventory, POS, Billing, Reports, User Management, and Subscription remain fully specified but should not destabilize the launch path.

Correction C - replace mixed screens with context-aware screens
Home, Profile, More, Settings, Reports, and Categories must behave differently by mode.
Do not hide architectural mixing under one generic screen title; use explicit context resolution.

Correction D - move from demo state to domain state
Current page-local arrays and local form state are good for design exploration but must be replaced by repositories, use cases, view models, and persistence layers.
No money, due, stock, or invoice values should remain hardcoded in production UI.

5. Canonical Auth, Onboarding, Account, and Context Flow
 
5.1 Final auth contract
• Login fields: phone number + password only.
• Signup fields: full name + phone number + password.
• OTP is not part of v2.1 auth unless the lock list changes.
• Forgot-password remains planned and hidden until both route and backend contract are ready.
• Google and Apple sign-in remain out of v2.1 launch scope.
Important build note
The current Auth UI must be reduced, not expanded. Production quality here comes from correctness and contract discipline, not from adding more auth buttons.

5.2 Startup and context resolution
Flow stage
Required outcome
Startup gate

No session

Session but missing profile/context

Personal context resolved
Route to Personal Home.

Route to Business Home.
Context incomplete or invalid

5.3 Post-auth selector flow
• After successful auth, show a context-safe selector if the user has multiple accounts or workspaces.
• If the user has no account yet, show Create Personal Account and Create Business Account as the first explicit split.
• Business account creation captures workspace metadata such as business name, business type, tax identity, address, currency, and settlement defaults.
• Personal account creation stays lean: display name, city/location, base currency, and optional personal finance defaults.

8. Personal Domain Build Specification
 
8.1 Personal Home
• Show income, expense, net movement, savings trend, upcoming reminders, and personal EMI due cards.
• Quick actions: Add Income, Add Expense, Transfer, Add Borrow / Lend, Pay EMI, View Reports.
• Do not show business receivable/payable widgets in Personal Home.
8.2 Personal Transactions
Micro-feature
Production requirement
List and filters
Date, type, category, account, contact, status, recurrence, search
Entry types
Income, expense, transfer, borrow, lend, repayment, recurring entry
Entry form
Account, category, amount, date, note, tags, attachment, reminder, contact
Detail screen
Audit trail, linked account, linked contact, linked installment if applicable
Recurring support
Template + next due logic; never duplicate silently

8.3 Personal accounts and contacts
• Personal accounts can include wallet, bank, cash, card, and savings-style holders.
• Personal contacts can include friend, family, borrower, lender, landlord, service provider, or institution.
• Opening balances and directionality must be explicit and never inferred from labels.
8.4 Personal EMI & Loans
• Use cases: phone EMI, laptop EMI, furniture EMI, scooter finance, personal bank loan, credit-card converted plan.
• Required views: plan list, plan detail, schedule, due card, pay installment, overdue state, closure state.
• All actual installment settlement must flow through InstallmentPaymentLink instead of ad hoc plan math.

9.3 Business contacts
• Customer and supplier are first-class business party types.
• A contact may hold balance direction, opening balance, tax identity, address, collector notes, and document links.
• Customer/supplier balances should be derived from ledger entries and settlements, not hand-edited summary labels.
9.4 Business EMI & Loans
• Use cases: equipment finance, asset loans, installment sales, business bank loans, structured collection plans.
• Business EMI can link to customer collections or business asset repayment depending on plan type.
• Plan type must stay explicit because the reporting, due wording, and linked documents differ.

10.2 Inventory
• Inventory owns stock on hand, stock movement, purchase in, sale out, return, adjustment, damaged/lost stock, and threshold alerts.
• Low-stock is an inventory rule. Notification delivery is shared infrastructure.
• Stock counts must be changed through explicit movement records, never only by editing a number on the UI.
10.3 POS
• The current slot-based POS shell is usable as a quick-sell prototype, but production POS must resolve active business and active settlement account before opening.
• POS needs cart, item search, category shortcuts, quantity change, discount, surcharge, tax, tender selection, partial payment, hold/void behavior, and receipt output.
• If the customer leaves unpaid balance, POS must create the correct ledger effect instead of silently treating every sale as paid.

11. Subscription, Localization, Tax, and Market Strategy
 
11.1 Market orientation
• Nepal-first and India-ready should remain the product design baseline for launch thinking.
• Global support should come from configuration and localization patterns, not from hardcoded Nepal-only or India-only assumptions.
• Language, currency, tax profile, numbering, and address format must be data-driven.
11.2 Language and locale policy
Layer
Requirement
Language
Support English, Nepali, and Hindi first. Bengali and additional languages can remain extension-ready if not launch-critical.
Currency
Base currency per account / business. Display support for NPR, INR, and global currencies.
Date/time
Use timezone-safe storage and locale formatting at presentation layer.
Text rules
Avoid business-only terminology in Personal mode. Avoid personal-finance wording in business-admin screens.

11.3 Subscription taxonomy
The current UI and the previous master document use conflicting subscription names. Production must choose one plan taxonomy and remove all duplicates. For consistency with the earlier master document, the recommended final stack is Platinum, Diamond, and Crown with entitlements stored in data, not hardcoded across screens.

11.4 Country tax and document policy
• Nepal profile needs PAN/VAT-aware invoice identity, configurable rates, invoice numbering policy, and e-billing readiness.
• India profile needs GSTIN support and configuration for GST invoice behavior including CGST / SGST / IGST routing.
• Global profile should default to a tax-profile abstraction rather than pretending one Nepal or India rule can fit every market.

• Profile 1:N Accounts.

13. Engineering, Testing, Security, Analytics, and Release Quality Gates
 
13.1 Architecture rules
• UI components stay dumb and mode-aware; they render state and dispatch callbacks.
• Page-local demo arrays must be replaced with repositories, use cases, and view models or equivalent application services.
• Do not let one mega screen own business logic across unrelated domains.
• Use shared modules for reusable capabilities, but keep domain ownership explicit.
• Plan naming, auth contract, and mode model must be centrally locked to avoid screen-by-screen drift.
13.2 Security rules
• Secrets, access tokens, and refresh tokens belong in secure storage only.
• No silent fallback to a random workspace, account, or settlement context.
• Permissions must be enforced at the application/service layer and not only hidden in the UI.
• Tax support is configurable product functionality, not legal advice.
• Financial edits and deletions that affect balances or plan links must be auditable.
13.3 Test gates
Test layer
Required coverage
Unit tests
Tax selection logic, plan schedule logic, overdue logic, threshold logic, entitlement checks
Repository tests
Mapping, null handling, relationship links, ownership scope, secure storage behavior
View model tests
Loading, form validation, happy path, edit path, delete path, denied path, mode switch path
UI tests
Auth, account setup, transaction form, ledger entry flow, plan payment, POS checkout, invoice save
Migration tests
Old records remain readable when ownership or link fields are introduced

13.4 Release gates
• No launch module ships without unit, repository, view model, UI, and migration coverage where applicable.
• No mixed-plan naming across UI and billing rules.
• No auth UI element may be visible if the route/API/backend contract is not launch-ready.
• No mock financial balances or chart data in production release builds.
• No layout should depend on hidden assumptions about the current mode.
