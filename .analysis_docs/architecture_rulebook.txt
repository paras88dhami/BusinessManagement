1. Non-negotiable architecture decisions
Every feature in eLekha must use one canonical request flow. The application may support personal 
mode and business mode, but the engineering contract must remain the same everywhere so features 
are predictable and easy to scale.
Canonical request flow
• Route -> Factory/Container -> Pure UI -> View Model -> Use Case -> Repository -> Data Source -> 
Database / API / Device Service
• No layer may skip around the stack. UI never imports repository or database. Use case never 
imports React. Data source never shows toast, alert, or UI strings.
 One business action must have its own use case. Do not group unrelated business actions into a 
single generic submit use case.
 One screen or one clear user intent must have its own view model. Do not build one giant view 
model for a whole module.
 Factories own dependency wiring and cross-feature composition. Routes own navigation only.
 Shared code must stay generic. If a file knows business words such as Transaction, Product, Ledger, 
Role, or EMI, it usually belongs in a feature, not in shared.
 Repository is the only place allowed to merge local, remote, cache, and mapping behavior for a 
feature aggregate.

4. Use case and view model granularity rules
The biggest enforcement rule is this: do not build one generic submit layer that tries to handle 
everything. Split by business action and by screen intent.

 List screen view model must not also contain add, update, delete, export, archive, and sync logic all 
in one file.
 If one screen has search, pagination, and filter state, those may be composed under the screen VM, 
but the business submit action must still live in its own use case.
 If two screens share the same business action but different UI state, they may reuse the same use 
case while keeping separate view models.
5. Exact rules for each layer
5.1 Route
 Route files may read params and decide which factory to render.
 Route files may pass navigation callbacks such as onBack, onSuccess, onOpenDetails, or 
onSwitchBusiness.
 Route files must not import repository, data source, database, API, secure storage, or use case 
directly.
5.2 Factory
 Factory creates the dependency graph: data source -> repository -> use case -> view model -> UI.\n\n--- PAGE 5 ---\neLekha Feature Architecture Rulebook
Version 1.0  |  5
 Factory is the only valid place to compose multiple feature view models when one screen depends 
on multiple modules.
 Factory must return JSX, not call a component function directly.
5.3 UI
 UI files are pure presentation. They accept props and callbacks only.
 UI may render table columns, forms, input states, modals, tabs, summaries, and empty states.
 UI must not format business rules that belong in the domain layer, such as EMI schedule generation, 
tax selection rules, settlement rules, or inventory adjustments.
5.4 View model
 View model owns screen state, form wiring, loading state, error state, effect orchestration, and 
intent handlers.
 View model converts use case outputs into UI-friendly state.
 View model may call multiple use cases but may not call database, Axios, or secure storage directly.
5.5 Use case
 Use case owns one business action and returns typed Result values.
 Use case validates inputs and coordinates repository methods.
 Use case must not know component lifecycle or React hooks.
5.6 Repository
 Repository exposes domain contracts. It maps models, merges sources, translates errors, and owns 
aggregate consistency.
 Repository is where local-first, remote-first, or sync merge policy lives.
 Repository must never return raw WatermelonDB models or raw Axios responses to UI.
5.7 Data source
 Data source performs raw IO only: DB query, network call, secure storage access, file access, or 
native service wrapper.
 Data source may be local, remote, memory, device, or mock.
 Data source must not coordinate business workflows across modules.
5.8 Types
 Any type imported by more than one file must leave the component file and live in types.
 Entity contracts, state contracts, error unions, DTOs, params, and response contracts belong in types 
files.
 Never use any or as any to skip typing discipline.
5.9 Shared
 Shared components must be generic, narrow, and feature-agnostic.
 Shared hooks must clean up timers, subscriptions, and listeners.
 Shared infrastructure must be typed strictly because every feature depends on it.
6. Type ownership standard
Type placement must be boring and predictable. The team should never guess where a contract lives.
 Entity and repository contracts: feature/types/<feature>.entity.types.ts
 UI state contracts and view-model output contracts: feature/types/<feature>.state.types.ts\n\n--- PAGE 6 ---\neLekha Feature Architecture Rulebook
Version 1.0  |  6
 Domain errors and Result discriminators: feature/types/<feature>.error.types.ts
 Transport input/output DTOs: feature/types/<feature>.dto.types.ts
 Shared Result, Status, AsyncState, PaginationState, SortDirection, and ID helpers: shared/types or 
shared/result.
 Tiny one-off prop types may stay in the same UI file only when they are local, trivial, and not 
imported elsewhere.
Type quality gate
• No new any or as any without a written temporary exception, owner, and removal date.
• Prefer exact unions, generics, unknown + narrowing, or explicit domain interfaces instead of 
loose casts.
7. Shared contracts every feature should use
export const Status = {
  Idle: "IDLE",
  Loading: "LOADING",
  Success: "SUCCESS",
  Failure: "FAILURE",
} as const;
export type StatusType = (typeof Status)[keyof typeof Status];
export type Result<T, E = Error> =
  | { success: true; value: T }
  | { success: false; error: E };
 Each use case should return Result<T, E> or a stricter domain equivalent.
 Each view model should expose explicit state objects, not many unrelated booleans spread across UI 
files.
 Feature errors must be mapped to domain-friendly messages and codes before they reach UI.
8. Hook, re-render, and crash-prevention rules
 Effects must include every reactive dependency they read unless the logic is intentionally 
restructured to avoid that dependency.
 Do not use useEffect to derive state that could be calculated during render or by useMemo.
 Do not wrap incoming callbacks with useCallback(fn, []). That freezes stale references.
 Use useMemo only for expensive derived values, stable identity for deep children, or controlled 
dependency graphs. Do not memoize everything.
 For tables and lists, use stable key extractors and optimize fixed-height rows when the data size 
justifies it.
 Factories must avoid churn by creating stable dependency instances instead of recreating the whole 
graph on every render.
Production review flags to reject in PR
• UI imports repository, datasource, database, secure store, or Axios.
• One giant submit use case handles add + update + delete + sync + extra actions.
• One giant view model owns the whole module instead of one screen or clear intent.\n\n--- PAGE 7 ---\neLekha Feature Architecture Rulebook

• Shared components or hooks contain product-, ledger-, transaction-, or role-specific wording.
9. Build blueprint for any new feature
 1. Define the aggregate root and entity contracts in feature/types.
 2. Define datasource contracts and local/remote implementations in feature/data/dataSource.
 3. Define repository contract and repository implementation in feature/data/repository.
 4. Add mapper file so persistence models and DTOs do not leak upward.
 5. Create one use case file per business action.
 6. Create one view model per screen or per intentful action flow.
 7. Build pure UI screens that only consume VM contracts and callbacks.
 8. Build the factory that wires all dependencies and passes navigation callbacks.
 9. Add the route file that renders the factory only.
 10. Add tests for use case success/failure, repository mapping/error translation, and view-model 
state transitions.

11. Cross-feature composition rule
A screen may use more than one module, but composition must happen in the factory. Example: a 
Billing form may need Contacts lookup, Products lookup, Tax calculation, and Accounts selection. The 
screen still receives a single clean UI contract, while the factory composes the necessary view models or 
service adapters.
 Do compose features in the factory/container.
 Do not import one screen into another and reach into its internals.\n\n--- PAGE 8 ---\neLekha Feature Architecture Rulebook

 Do not let UI call multiple repositories across features.
 When cross-feature actions exist, create a dedicated orchestration use case with a clear boundary 
and result type.
12. Testing minimum for every feature
 Use case tests: success path, validation failure path, repository error path.
 Repository tests: datasource mapping, local/remote merge, null handling, and error translation.
 View model tests: loading, empty, success, failure, edit hydration, and submit state reset.
 UI tests: renders loading, empty, error, and happy path callbacks correctly.
 Factory tests: when composition is complex, verify the right dependencies are wired and the right 
callbacks are injected.
Definition of done
• No new loose typing, no direct infrastructure import in UI, no stale hook dependencies, no 
merged business actions in one generic use case, and no feature-specific logic inside shared.
• Every CRUD action must have a dedicated use case. Every screen or intent flow must have a 
dedicated view model.
13. Pull request checklist
☐ Review item
☐ No new any or as any unless there is a documented exception with owner and removal 
date.
☐ All effects have exact dependency arrays or were rewritten to avoid unstable dependencies.
☐ No UI file imports a repository, datasource, database instance, secure storage, or transport 
utility directly.
☐ Factories create stable dependency graphs and render JSX instead of invoking component 
functions directly.
☐ Repository owns mapping and error translation. Datasource remains dumb.
☐ Each business action has its own use case. Each screen/intent has its own view model.
☐ Shared code remains generic and feature-agnostic.
☐ Tests cover success and failure paths for shipped actions.
14. Final enforcement statement
This rulebook is the build contract for eLekha features. The team should not treat it as optional 
guidance. A feature is not production-grade unless its route, factory, UI, view model, use case, 
repository, datasource, types, and shared boundaries match this standard.
Simple memory rule for the team
• Route chooses. Factory wires. UI renders. View model orchestrates. Use case decides. Repository 
translates. Datasource performs IO. Shared stays generic.
